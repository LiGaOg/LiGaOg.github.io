通过分页实现虚拟内存

一级页表

首先, 对于物理内存来说, 它被分成了很多的pages.

  • Page size最典型的大小是4KB, 也就是$2^{12}$ B.

如果我要定位一个page中的某个字节, 我需要两个数值:

  • 这个Physical Page的编号, 也可以叫Physical Page Number (PPN).
  • 这个字节在Page中的Offset.

这个Offset一般被编码在虚拟地址上.

对于PPN, 一般存储在页表(Page Table)的页表项(Page Table Entry)中.

如果我要定位页表中某一个页表项, 那么同样需要两个数值:

  • Page Table的起始地址所在虚拟页的虚拟页编号(VPN), 注意是编号, 不是实际的物理地址.
  • PTE在Page Table中的Index.

对于这个Index, 同样被编码在虚拟地址上, 也叫做Virtual Page Number (VPN), 因为一个PTE用来管理一个虚拟页面.

对于Page Table的起始地址, 一般存储在CSR寄存器中.

因此, 对于一级页表, 它的Page Table Walk算法的步骤是:

  • 首先, 从寄存器中读取Page Table的起始地址.
  • 然后, 根据起始地址和VPN, 定位到PTE, 读取到PPN.
  • 最后, 根据PPN以及Offset定位到某个字节.

多级页表

多级页表中, 除了最后一级页表的PTE存储Physical Page的起始地址, 其他页表的PTE都存储下一级页表的起始地址.

为什么需要多级页表?

本质原因是, 一个进程其实占用不到多少Physical Page, 但是如果采用一级页表, 一旦这个进程占用了一点Physical Page, 那么它对应的一级页表就要被创建出来, 一级页表其实很大, 系统又有很多进程, 这样物理内存就会被耗光.

如果采用多级页表, 能够有如下两个因素减少内存占用:

  • 一级页表的VPN减少, 一级页表如果创建, 它的PTE的数量可以减少.
  • 对于后续的页表, 只有在进程用到对应的地址的时候, 对应页表才会创建, 也就是说, 一级页表是每个进程都有一个, 但是后续的页表是否有, 是和进程访问的地址相关的.

更本质的原因, 其实就是内存访问的空间局部性(Spatial Locality).

TLB的体系结构

首先, TLB本质上也是一个缓存, 它也可以是Set Associative Cache.

对于一个虚拟地址, 它可以分成VPN和Offset (如果是多级页表VPN还可以分成VPN2, VPN1, VPN0).

对于VPN来说, 它可以直接分成Index和Tag, 根据Index和Tag可以定位到对应的TLB表项并判断是否命中.

如果命中了, 就可以直接拿到PPN, 结合Offset, 就可以直接在物理内存中取出数据.

Page Table的填充过程

注意, Page Table Walk是硬件实现的, 但是页表的初始化和填充是软件实现的.

  • 首先, 一个进程一旦创建, OS就会在物理内存中分配一级页表.

  • 当这个进程访问内存时, 它去一级页表看, 发现对应的PTE的valid bit是0, 触发Page Fault.

  • OS接管, 为其分配并初始化二级页表, 并把二级页表的起始地址写入一级页表PTE中.

  • 然后分配三级页表, 并把三级页表的起始地址写入二级页表PTE.
  • 然后重复上述过程…
  • 最后在物理内存中分配一个Physical Page, 写入最后一级页表的PTE.

Demand Paging和Page Fault

如果采用虚拟内存, 那么一个进程有很大的虚拟地址空间, 实际上的物理内存肯定是不够存的.

例如, 假设一个系统有多个进程, 一个进程给了非常多的Physical Pages, 物理内存都不够了, 那么就需要将若干的Physical Page放到磁盘中 (选哪个Physical Page放到磁盘需要特殊策略).

放入磁盘之后, 对应最后一级页表的PTE要做标注, 说明这个Physical Page在磁盘中.

当进程在访问这个Physical Page的时候, 会触发Page Fault, OS会接管, 并且把这个Physical Page从磁盘搬到对应的物理内存中.

这个过程叫做Demand Paging.

Cache与虚拟内存

如果启用了虚拟内存, 那么对于Cache来说, 有两种处理方法:

  • 先让MMU/TLB把虚拟地址转换成物理地址, 然后用物理地址索引缓存.
    • 缓存需要等待MMU/TLB工作完才能工作, 增加了流水线延迟.
  • 直接用虚拟地址索引缓存.

如果使用虚拟地址索引缓存, 那么理想的工作流程应该是这样的:

  • CPU把虚拟地址给Cache, 看Cache是否命中.
    • 命中了就不用MMU/TLB.
  • Cache没命中, 交给MMU/TLB, 获得物理地址.
  • 根据物理地址访问内存.

一般来说, 如果使用虚拟内存索引缓存的话, 有两种实现方式:

  • VIVT (Virtual Index Virtual Tag): 直接用虚拟地址求Index和Tag来访问Cache.
  • VIPT (Virtual Index Physical Tag): 使用虚拟地址求Index, 使用物理地址求Tag.
    • 使用虚拟地址的Index和Offset索引Cache Line的同时, 把虚拟地址交给MMU/TLB进行翻译, 同时进行.
    • 最后判断是否Cache Hit使用物理地址的Tag.

虚拟内存的一些例题

例题1-一级/多级页表

假设进程的虚拟地址是32位 (4GB), 页的大小是4KB, PTE的大小是4字节.

  • Page Offset有多少位?
    • 页的大小是4KB = 2^{12}B, 那么Page Offset就有12位.
  • Page Number有多少位?
    • 一共有2^{32} / 2^{12} = 2^{20}个页.
    • 那么Page Number有20位.

假设现在只有一级页表 (linear page table), 目前进程只有一个physical page被建立映射.

  • 有多少valid PTE?
    • 就一个.
  • Page Table的大小是多大?
    • 一共有2^{20}个PTE, 每个PTE的大小是4B, 那么page table的大小就是4MB.

现在假设有二级页表, 两个页表对应的VPN的比特数是一样的(都是10位), 现在进程只有一个physical page被建立映射.

  • 有多少valid PTE?
    • 有两个, 一级页表和二级页表各有一个.
  • Page Table的大小多大?
    • 首先, 对于一级页表, 一共有2^10个PTE, 每个PTE的大小是4B, 那么总大小就是4KB.
    • 对于二级页表, 因为只有1个physical page被映射了, 所以只有一个二级页表, 总大小也是4KB.
    • 最后, 总大小就是8KB.

因此, 多级页表比二级页表更容易减少内存.
