为什么需要Cache

  • Spatial Locality: CPU访问了内存的一个地址, 它还会访问这个地址周围的地址.

  • Temporal Locality: CPU再某个时间访问了一个地址, 它很有可能在最近还访问.

  • Cache一般集成在CPU内部的SRAM中, 如果cache命中, 可以以更少的时钟周期拿到数据.

衡量Cache性能的指标

  • AMAT (Average Memory Access Time)
    • AMAT = HIT TIME + MISS RATE * MISS PENALTY
  • 因此, 有三种方法优化Cache的性能:
    • 减少Hit time.
      • 用更快更小的cache.
    • 减少Miss rate.
      • 用更大的cache, cache的策略选的好一点.
    • 减少Miss penalty.
      • 减少main memory的访问速度.

Cache的架构

N-way Set Associative Cache

可以把Cache理解成一个二维数组:

  • 每一行叫做一个Set.
  • 每一列叫做一个Way.
  • 一行一列的元素叫做一个Cache Line.

对于一个N-way Set Associative Cache, 它的列数就是N.

那么如何计算对于N-way Set Associative Cache来说, 一个地址怎么找到对应的Cache Line呢?

首先, 假设Cache Line的大小是b个字节, 那么地址最后的log2(b)位就是Offset.

之后, 我需要计算有几个Set, 也就是有几行?

  • 假设Cache总大小是C个字节, 那么Set的个数就是C / (b * N).
  • 那么Index就是log2(C / (b * N))位.

剩下的位就是Valid bit, Dirty bit以及Tag bits.

地址映射的时候, 先根据Index找到Set的编号, 然后同时比较地址的Tag和所有Way的Tag, 如果Tag一样并且Valid bit是1, 那么就是Cache hit.

Direct-Mapped Cache

Directed-Mapped Cache相当于Way是1的Set Associative Cache (列数是1).

Fully-Associative Cache

Fully-Associative Cache相当于Set数是1的Set Associative Cache (函数是1).

Cache Replacement Policy

Cache Replacement处理的问题是:

  • 如果Set Associative Cache中的一个Set满了, 这时候又要进来一个Cache Line, 那么这个Cache Line要放到哪个位置?

有如下几种算法:

  • Random: 随机一个Cache Line出去.

  • FIFO: 最先进来的Cache Line出去.

  • LRU: 让最久没被使用过的Cache Line出去.
  • MRU: 让最近使用过的Cache Line出去.
    • NMRU:
      • 首先记录一下谁是最近使用过的Cache Line (和MRU相似)
      • 剩下的Cache Line随机一个出去.

Cache Miss的来源

  • Compulsory Miss:

    • 第一次访问肯定Cache Miss, 因为Cache里面就没有东西.
    • 改进: 增加Cache Line Size, 但是Conflict Miss会增加, 并且Miss Penalty会增加.

  • Capacity Miss:

    • Cache太小了, 你要的Cache Line已经被请出去了.
    • 改进: 增加Cache Size, 但是Hit Time会增加.

  • Conflict:

    • Conflict的精准定义是: 如果这个Miss在Fully Associative Cache中不会发生, 那么这个Miss就是Conflict Miss.
    • Conflict Miss和Cache replacement policy相关.
    • 改进: 增加Cache Size和Associativity, 但是会增加Hit Time.

Cache与内存写操作

假设我现在要写一个内存地址, 假设写入的地址有Cache Hit, 那么Cache有两种策略处理:

  • Write through: 直接在Cache和main memory中同时修改即可.
  • Write Back: 现在Cache中写, 然后当这个Cache Line被evicted出去的时候再同步到Memory中.
    • 如果采用Write Back策略, 那么就需要在Cache Line中增加一个Dirty Bit, 标识这个Cache Line是不是被写过.

Write Buffer

当你采用Write Back策略的时候, 如果一个被修改过的Cache Line被evicted出去了, 那么这个修改就要被同步到main memory中.

这个时候, 如果你有一个读内存的请求 (cache miss了, 需要访问main memory), 并且这个请求的地址和刚才修改的cache line毫不相关, 那么这个读内存的请求就得等刚才evicted cache line写完内存才能开始, 这就很浪费时间.

这个时候, 我可以把这个cache line放到一个write buffer中, 之后再让write buffer中的东西同步到main memory中, 放到write buffer很快, 那么刚才读内存的请求就可以很快得到响应.

并且还有一个好处: 如果读内存正好读到了Write Buffer中的Cache line, 就可以直接将读到的数据bypass, 这个可以减少Miss Penalty.

Cache的一些例题

例题1-Cache Organization

有一个1KB大小的, 4-way的Set Associative Cache, Cache line的大小是32B, 地址是12bits, 请问地址的bits怎么划分成Tag, Index和Offset?

  • 怎么找Offset ?

    • Cache line的大小是32B, 那么Offset就有5位.
  • 怎么找Index?
    • 4-way表示一个Set中有4个Cache line.
    • 那么一个Set的大小就是32B * 4
    • 那么总共的Set的个数就是1024B / (32B * 4) = 2^{10}B / 2^{7}B = 2^{3}.
    • 那么Index就有3位.
  • 剩下的最后4位就是Tag.

  • Sub-blocks(sector cache)是什么意思?
    • 把cache line分成若干个小块, 每次Miss只从memory中取其中的小块, 就不用取整个cache line.
  • Compiler Prefetching是什么意思?
    • 假设你的Compiler知道你要做矩阵乘法, 那么他就可以插入一些指令让Cache充满你的矩阵的数据.

例题4-Prefetch

看如下这段代码:

int A[32][32];
int sum = 0;

for (int j = 0; j < 32; j ++) {
  for (int i = 0; i < 32; i ++) {
    // Prefetch from A + M * i + j + OFFSET
    prefetch(&A[i][j] + OFFSET);
    sum += A[i][j];
  }
}

假设cache line的大小是128B, 如果没有prefetch, 那么sum += A[i][j]需要耗费50个cycles, 其中40cycles浪费在了L1 cache miss上, 10cycles花费在加法上. 现在加上了prefetch, 请问OFFSET设置成多少, 才能够让循环所需要的cycle数目最小?

  • 首先, 加法需要10cycles, prefetch的数据返回需要40cycles.
  • 那么, 我可以在每次循环开始前, 都prefetch四次循环的数据, 然后再运行四次循环, 这样每次循环的平均cycle数目就是(40 + 10 * 4) / 4 = 20.
  • 那么Offset就是32 * 4 = 128.
